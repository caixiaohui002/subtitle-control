## 🚀 优化内容

### 1. 重构处理流程

**原流程（KY001）**：
1. 删除报幕信息
2. 按换行符分割成行
3. 对每行按标点符号拆分
4. 对超过字数的行使用 LLM 语义拆分

**新流程（KY002）**：
1. ✅ 删除报幕信息
2. ✅ 按标点符号拆分（直接对全文操作）
3. ✅ **新增**：删除所有标点符号（得到无标点版本）
4. ✅ 首轮检测并拆分超长行（批量收集超长行）
5. ✅ **新增**：通篇检查，确保没有超长行（二轮验证）

### 2. 性能优化

#### 批量处理超长行
- 收集所有超长行，统一处理
- 减少不必要的进度更新
- 提高处理效率

#### 优化进度更新
- 步骤1：检测时每5行更新一次
- 步骤2：拆分时每2行更新一次
- 步骤3：检查时每10行更新一次
- 减少前端渲染压力

#### 二轮验证机制
- 首轮拆分后，通篇检查所有行
- 发现仍有超长行，再次拆分
- 确保最终结果100%符合字数要求

### 3. 详细调试日志

**新增调试信息**：
- 按标点拆分后行数
- 删除标点后行数
- 首轮检测发现的超长行数量
- LLM 拆分的详细记录（每行拆分前20字 + 拆分结果）
- 通篇检查发现的超长行数量
- 二轮拆分的详细记录
- 最终验证结果（通过/警告）

## 🔧 技术改进

### 1. 流程优化
```typescript
// 旧流程：逐行处理
for (let i = 0; i < lines.length; i++) {
  if (charCount > maxChars) {
    await splitWithLLM(line, maxChars);
  }
}

// 新流程：批量收集
const longLines = [];
for (let i = 0; i < lines.length; i++) {
  if (charCount > maxChars) {
    longLines.push({ index: i, line });
  }
}
// 批量处理
for (const { line } of longLines) {
  await splitWithLLM(line, maxChars);
}
```

### 2. 二轮验证
```typescript
// 通篇检查
const secondRoundLongLines = [];
for (let i = 0; i < finalLines.length; i++) {
  if (countChineseChars(finalLines[i]) > maxChars) {
    secondRoundLongLines.push({ index: i, line });
  }
}

// 再次拆分
for (let i = secondRoundLongLines.length - 1; i >= 0; i--) {
  const { index, line } = secondRoundLongLines[i];
  const splitLines = await splitWithLLM(line, maxChars);
  finalLines.splice(index, 1, ...splitLines);
}
```

### 3. 验证机制
```typescript
// 验证所有行都不超过字数
let allValid = true;
for (let i = 0; i < finalLines.length; i++) {
  if (countChineseChars(finalLines[i]) > maxChars) {
    sendDebug(`警告：第 ${i + 1} 行仍超长...`);
    allValid = false;
  }
}
if (allValid) {
  sendDebug(`验证通过：所有行均≤${maxChars}字`);
}
```

## 🎯 工作流程（KY002）

### 步骤1：删除报幕信息
- 识别并删除报幕行

- 收集所有超长行
- 批量 LLM 语义拆分

### 步骤5：通篇检查
- 检查所有行字数
- 发现超长行则再次拆分
- 最终验证所有行符合要求
- 输出验证结果

## ⚠️ 注意事项

1. **二轮拆分会增加处理时间**
   - 如果 LLM 拆分质量高，通常不需要二轮
   - 二轮仅作为兜底机制

2. **调试日志更详细**
   - 生产环境可以考虑关闭部分调试日志
   - 或降低日志级别

3. **性能提升主要来自批量处理**
   - 超长行越多，性能提升越明显
   - 短文本性能提升不明显
